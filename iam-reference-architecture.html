Problem

IAM tools need a shared architecture.

Identity environments often contain Entra ID, Active Directory, SaaS apps, service desk workflows, privileged roles, and monitoring tools. Without a reference architecture, controls can become disconnected and hard to govern.

Solution

A target-state model for identity governance.

This project defines how identities enter, move through, and exit the environment while tying authentication, authorization, review evidence, and monitoring into a single IAM operating model.

Architecture layers

Target-state IAM model

Identity Source

HR record, approved request, or service desk ticket starts the lifecycle event.

Directory Layer

Microsoft Entra ID and Active Directory hold users, groups, roles, and account state.

Access Layer

Groups, roles, app assignments, and SSO integrations grant business access.

Control Layer

MFA, Conditional Access, privileged access practices, and least privilege reduce risk.

Governance Layer

Access reviews, tickets, approvals, audit logs, and exception handling prove control.

Design matrix

Architecture decisions

Lifecycle

Joiner, mover, and leaver flows are tied to approved triggers and evidence capture.

Access Model

Prefer RBAC groups, app roles, and standard access packages over direct assignment.

Privileged Access

Separate administrative accounts, protect emergency access, and review privileged roles.

Monitoring

Use sign-in logs, audit logs, SIEM alerts, and incident handoffs to detect identity risk.

Practice steps

Hands-on architecture path

  1. Define business goals and identity risk drivers.

  7. Create a phased roadmap from current state to target state.
