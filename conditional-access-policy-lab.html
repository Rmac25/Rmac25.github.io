Project case study

Conditional Access Policy Lab

A Microsoft Entra ID lab focused on designing, validating, and safely rolling out MFA-based Conditional Access policies for cloud applications.

Download Runbook Back to Portfolio
Platform
Microsoft Entra ID
Controls
MFA, Conditional Access, exclusions
Validation
Report-only mode + sign-in logs
Security Goal
Least privilege access enforcement

Problem

MFA policies can fail when rollout is rushed.

Conditional Access is powerful, but a poorly scoped policy can interrupt business users, lock out administrators, or create unclear troubleshooting paths. IAM teams need a safe design that protects access without creating avoidable outages.

Solution

A staged policy design with validation evidence.

This lab maps policy requirements to Entra ID assignments, exclusions, grant controls, report-only validation, and sign-in log review before enforcement.

Policy architecture

How the lab is structured

01

Scope

Target a pilot security group first, define cloud applications, and exclude approved emergency access accounts.

02

Control

Require multifactor authentication for selected users and applications while documenting why the control is needed.

03

Validate

Use report-only mode and sign-in logs to confirm expected outcomes before enabling enforcement.

Design matrix

Conditional Access design decisions

Assignments

Target pilot users through a security group instead of immediately applying the policy tenant-wide.

Exclusions

Exclude documented break-glass accounts so emergency access remains available during policy issues.

Cloud Apps

Start with Microsoft 365 and selected SaaS apps, then expand after validation.

Grant Controls

Require MFA, confirm registration status, and review interruptions before enabling enforcement.

Practice steps

Hands-on lab path

Download Full Runbook
  1. Create a pilot security group and add a test user.
  2. Confirm MFA registration and document break-glass exclusions.
  3. Create a Conditional Access policy scoped to the pilot group.
  4. Select Microsoft 365 or another test cloud application.
  5. Require MFA and start the policy in report-only mode.
  6. Test browser and mobile sign-ins, then review Entra ID sign-in logs.
  7. Document expected results before moving the policy to enforced mode.

Validation evidence

What I would verify before enforcement

The lab emphasizes report-only validation and sign-in log review so the policy can be explained, defended, and safely moved into production.

Validation checklist
- Target user is included in the pilot group
- Break-glass account is excluded
- Policy appears in report-only sign-in results
- Grant control shows MFA requirement
- Browser and mobile sign-ins behave as expected
- Interrupted sign-ins are documented
- Enforcement is approved through change management
